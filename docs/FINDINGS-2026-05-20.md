## 1. P0 — Must Fix Before Demo

### P0-0: `generate-campaign-brief` — Intermittent `brief_parse_failed` (JSON Parse Error) — **FIXED**

**File:** `supabase/functions/generate-campaign-brief/index.ts`

When generating a campaign brief, `deepseek/deepseek-v4-flash` intermittently returns malformed JSON that `JSON.parse()` fails on, producing a `brief_parse_failed` error. The LLM can wrap output in thinking/reasoning text, code fences, or produce truncated JSON.

**Fix applied:**
1. `extractJSONObject()` — robust JSON extraction that tries: direct parse → markdown fence extraction → `{...}` slice with trailing-comma fix
2. Retry up to 2 attempts — if first LLM call produces unparseable output, re-invoke the LLM
3. Better error diagnostics — `detail` and `raw_preview` in error response

### P0-0b: `generate-campaign-brief` — Schedule Dates Ignore `start_date` and `working_days_only` — **FIXED**

**File:** `supabase/functions/generate-campaign-brief/index.ts`

The prompt used `new Date().toISOString().slice(0, 10)` as the start date instead of the campaign's `start_date` field. This caused the LLM to generate posts for today's date rather than the actual campaign start (e.g., May 23 Sat/Sun entries for a campaign starting May 27).

Additionally, the LLM was only told "skip Saturday & Sunday" without being given the exact dates, so it miscalculated working-day dates (e.g., placing day 1 and 2 on a Saturday and Sunday).

**Fix applied:** Compute the exact list of working-day dates from `start_date` + `duration_days` + `working_days_only` and pass them directly to the prompt: "Use these EXACT working-day dates for the schedule entries."

The prompt said "Generate at least 2 variants per channel" but a 7-day campaign needs 7 posts (one per day). The LLM returned only 2 `linkedin_post` entries, leaving days 2-7 without copy.

### P0-2: `save-onboarding` — Arbitrary Body Upserted Into Database

**File:** `supabase/functions/save-onboarding/index.ts:66-78`

```ts
const body = await req.json()
const { complete, ...brandPayload } = body
const upsertData: Record<string, unknown> = {
  org_id,
  ...brandPayload,    // entire client payload spread into DB write
  updated_at: new Date().toISOString(),
}
```

Three problems:

1.  **No field allowlist.** The raw client payload is spread into the `brand_contexts` upsert. A malicious or buggy client can inject arbitrary columns (e.g., overwriting `id`, setting `org_id` to a different org — though `org_id` is correctly set from JWT on the line above, the rest of the payload is uncontrolled).
2.  **No Zod validation.** A `SaveOnboardingBodySchema` exists in `_shared/schemas.ts` but is not used here.
3.  **No `requireRole()` check.** Any authenticated user, including `member` role, can modify org onboarding data. All other mutation endpoints restrict to `admin` or `owner`.

**Impact:** Mass assignment vulnerability. Any column in `brand_contexts` can be overwritten by any authenticated user.

**Fix:**

1.  Validate the body with `SaveOnboardingBodySchema.safeParse(rawBody)`.
2.  Add `await requireRole(org_id, user.id, 'admin', db)` after extracting the org_id.
3.  Construct `upsertData` from explicitly-named validated fields, not from the spread payload.

### P0-3: 6 Functions Read `org_id` From JWT Without UUID Format Validation

**Files and lines:**

| Function | File | Line |
|---|---|---|
| `generate-asset` | `generate-asset/index.ts` | 36 |
| `check-quota` | `check-quota/index.ts` | 12 |
| `build-prompt` | `build-prompt/index.ts` | 57 |
| `submit-feedback` | `submit-feedback/index.ts` | 12 |
| `get-upload-url` | `get-upload-url/index.ts` | 57 |
| `save-onboarding` | `save-onboarding/index.ts` | 54 |

All six read `org_id` as:

```ts
const org_id = user.app_metadata?.org_id as string | undefined
```

The shared `extractOrgId(user)` function in `_shared/auth.ts` validates that the value is a well-formed UUID and throws a 401 if it is not. These six functions bypass that validation. If `app_metadata.org_id` were ever malformed (empty string, non-UUID), it would be passed directly into Supabase queries.

All other user-facing functions correctly use `extractOrgId(user)`.

**Impact:** Inconsistent org_id validation. A malformed `org_id` claim would pass through to database queries instead of being rejected at the edge.

**Fix:** Replace `user.app_metadata?.org_id as string | undefined` with `extractOrgId(user)` in all six functions.

### P0-4: Encryption `btoa()` Stack Overflow on Large Payloads

**File:** `supabase/functions/_shared/encryption.ts:34`

```ts
return btoa(String.fromCharCode(...combined))
```

The spread operator `...combined` passes every byte as a separate argument to `String.fromCharCode`. When the encrypted payload (IV + ciphertext) exceeds approximately 65,536 bytes — which can happen for large BYOK API keys or encrypted LinkedIn access tokens — the function throws:

```
RangeError: Maximum call stack size exceeded
```

The `decrypt` function (line 45) correctly uses `Uint8Array.from(atob(ciphertext), c => c.charCodeAt(0))` which avoids this issue.

**Impact:** Any encryption operation on a payload >65KB will crash, returning a 500 error. BYOK key saves or LinkedIn token saves of moderate length could hit this.

**Fix:** Replace line 34 with:

```ts
return btoa(Array.from(combined).map(b => String.fromCharCode(b)).join(''))
```

Or use the same pattern as `decrypt`:

```ts
let binary = ''
for (let i = 0; i < combined.length; i++) {
  binary += String.fromCharCode(combined[i])
}
return btoa(binary)
```

### P0-5: `campaign-chat` — Orphaned User Message on LLM Failure

**File:** `supabase/functions/campaign-chat/index.ts:181-185`

The user's message is inserted into `campaign_chat_messages` at line 182 **before** the LLM call at line 206+. If the LLM call fails (line 271 returns a 502), the user's message already exists in the database with no corresponding assistant reply.

On the next chat request, the history query (line 192) includes this orphaned user turn, which corrupts the context window — the LLM sees a user message with no response, leading to confused or degraded output.

**Impact:** After any LLM failure, the chat becomes degraded for that campaign. Multiple failures accumulate orphaned messages.

**Fix:**

Option A (preferred): Move the user message insert to after the LLM response succeeds.

Option B: In the error/catch path, delete the user message that was just inserted:

```ts
await db.from('campaign_chat_messages').delete().eq('id', insertedMessageId)
```

### P0-6: `generate-asset` — TOCTOU Race Condition on Quota Deduction

**File:** `supabase/functions/generate-asset/index.ts`

The quota check at line ~95 reads:

```ts
const imageLimit = org?.plan_tier === 'fully_subscribed' ? org?.image_limit : starterQuota.image
const imageUsed = org?.image_used ?? 0
if (imageUsed >= imageLimit) { return 402 }
```

The quota is deducted later at lines 251 or 286-289:

```ts
await db.from('orgs').update({ image_used: (org?.image_used ?? 0) + 1 }).eq('id', org_id)
```

This is a read-then-write pattern. Under concurrent requests from the same org, two requests can both read `image_used = 49`, both pass the `>= 50` check, and both write `image_used = 50`. The result: 51 images used against a 50-quota.

The service-role client bypasses RLS, so there is no database-level guard against this.

**Impact:** Concurrent image/video generation requests can exceed the plan quota by 1 per concurrent request.

**Fix:** Use an atomic SQL update via an RPC call:

```sql
-- New migration
CREATE OR REPLACE FUNCTION increment_image_used(p_org_id uuid)
RETURNS void AS $$
  UPDATE orgs SET image_used = image_used + 1 WHERE id = p_org_id;
$$ LANGUAGE sql SECURITY DEFINER;
```

Or add a WHERE clause guard for optimistic concurrency:

```ts
const { count } = await db.from('orgs')
  .update({ image_used: org.image_used + 1 })
  .eq('id', org_id)
  .eq('image_used', org.image_used)  // CAS-style guard
```

### P1-1: `personalise` — Wrong Column Names in Model Fallback Query

**File:** `supabase/functions/personalise/index.ts:102-108`

```ts
const { data: defaultModel } = await db
  .from('available_models')
  .select('provider_key, model_id')
  .eq('step_key', 'outreach_copy')    // column does not exist
  .eq('is_default', true)             // column does not exist
  .single()
```

The `available_models` table does not have columns named `step_key` or `is_default`. The correct columns are:

- `default_for_step_key` (an array column, queried with `.contains()`)
- `is_active` (a boolean)

Other functions (`campaign-chat`, `generate-captions`, `build-prompt`) correctly use:

```ts
.contains('default_for_step_key', ['outreach_copy'])
```

**Impact:** Every `personalise` call that doesn't have an org preference set silently falls through to the hardcoded `google/gemini-2.5-flash` on OpenRouter, ignoring the configured default model. This means:

- Model quality for personalisation may differ from what's configured in settings.

**Fix:**

```ts
const { data: defaultModel } = await db
  .from('available_models')
  .select('provider_key, model_id')
  .contains('default_for_step_key', ['outreach_copy'])
  .eq('is_active', true)
  .maybeSingle()
```

### P1-9: Cron Functions — Service Role Key Compared via `.includes()`

**Files:**

- `ingest-signals/index.ts:10`
- `archive-old-signals/index.ts:10`
- `cleanup-apify-signals/index.ts:10`
- `poll-job-status/index.ts:101`
- `reset-monthly-quotas/index.ts:10`

```ts
authHeader.includes(serviceRoleKey)
```

This is a substring match on the full Authorization header. While practically unlikely to match a different string, `.includes()` is not the right comparison operator for a secret:

- If the service role key is a substring of some other string in the header, it would match.
- If future code adds `Bearer` prefix or other formatting, `.includes()` would silently match while `===` would fail (a safer failure mode).

**Fix:** Use exact comparison:

```ts
authHeader === `Bearer ${serviceRoleKey}`
// or for consistency with other cron functions that use X-Cron-Secret:
```

### P1-DB-3: 4 RLS Policies Use Raw JWT Path Instead of `current_org_id()`

Migration `0012` introduced the `current_org_id()` helper function (which uses `nullif(coalesce(...), '')::uuid` for NULL-safe handling) and rewrote all existing RLS policies to use it. However, tables added after migration 0012 regressed to the raw JWT claim path:

| Table | Migration | Claim Path Used | `current_org_id()`? |
|---|---|---|---|
| `refinement_options` | 0016 | `auth.jwt() -> 'app_metadata' ->> 'org_id'` | No |
| `org_linkedin_connections` | 0021 | `auth.jwt() -> 'app_metadata' ->> 'org_id'` | No |
| `campaign_chat_messages` | 0021 | `auth.jwt() -> 'app_metadata' ->> 'org_id'` | No |
| `org_chat_usage` | 0021 | `auth.jwt() -> 'app_metadata' ->> 'org_id'` | No |
| `icp_enrichment_runs` | 0024 | `public.current_org_id()` | Yes |

The `current_org_id()` helper safely handles:
- Empty string org_id (returns NULL → RLS evaluates to false → no data leak)
- Missing `app_metadata.org_id` (falls back to top-level `org_id` claim)
- Invalid UUID format (returns NULL → RLS evaluates to false)

The raw path `auth.jwt() -> 'app_metadata' ->> 'org_id'` does none of these.

**Fix:** Write a migration that replaces the 4 policies to use `current_org_id()`. Example for `campaign_chat_messages`:

```sql
CREATE OR REPLACE POLICY org_isolation_select ON public.campaign_chat_messages
  FOR SELECT USING (org_id = public.current_org_id());
-- Repeat for INSERT, UPDATE, DELETE
```

### P1-DB-4: `SECURITY DEFINER` Functions Missing `search_path`

**Files:**

- `0002_rls_policies.sql:7` — `auth_org_id()` defined as `LANGUAGE sql STABLE SECURITY DEFINER` without `SET search_path`.
- `0006_usage_and_storage.sql:27` — `increment_org_quota()` defined as `LANGUAGE plpgsql SECURITY DEFINER` without `SET search_path`.

By contrast, `set_refinement_options_updated_at()` in migration `0016` correctly sets `SET search_path = public, pg_temp`.

PostgreSQL best practice (and Supabase's own hardening guide) requires all `SECURITY DEFINER` functions to set `search_path` to prevent search-path injection attacks.

Additionally, `auth_org_id()` was replaced by `current_org_id()` in migration `0012` but was never dropped. It is dead code that could confuse future developers or accidentally be referenced.

**Fix:**

1.  Add `SET search_path = public, pg_temp` to `increment_org_quota()`.
2.  Drop `auth_org_id()` since all policies now use `current_org_id()`.

```sql
ALTER FUNCTION public.increment_org_quota() SET search_path = public, pg_temp;
DROP FUNCTION IF EXISTS public.auth_org_id();
```

```ts
const liCache = new Map<string, LiCacheEntry>()
```

The cache grows without bound. Stale entries are checked for TTL but never removed from the Map, causing slow memory growth in the Deno isolate.

**Fix:** Add periodic eviction based on `fetchedAt`, or use an LRU cache with a size limit.

### P2-2: 5 Functions Use Inline CORS Instead of Shared Module

**Files:** `accept-invite`, `create-org`, `get-upload-url`, `ingest-signals`, `save-onboarding`

These five functions implement CORS handling inline rather than importing from `_shared/cors.ts`. The inline implementations lack the `Access-Control-Max-Age` header and may drift from the canonical allowlist over time.

**Fix:** Import and use `handleCors(req)` / `getCorsHeaders(req)` from `_shared/cors.ts`.

### P2-3: User Data Injection Surface in Campaign Brief Prompt

**File:** `supabase/functions/generate-campaign-brief/index.ts:~500-870`

The prompt template injects user-controlled data (brand names, competitor names, campaign descriptions) directly into the LLM prompt without escaping. A maliciously crafted `description` or `goal` could contain instruction-like text.

**Fix:** Add a heuristic check that user-provided strings don't contain newlines starting with `#` (markdown headings which could be confused with prompt sections).

### P2-4: Module-Level Cache Map in `get-available-models` Never Pruned

**File:** `supabase/functions/get-available-models/index.ts:7-8`

```ts
const cache = new Map<string, { data: any; timestamp: number }>()
const CACHE_TTL_MS = 10 * 60 * 1000
```

Stale entries are checked for TTL on read but never deleted from the Map. With a small number of orgs this is fine, but at scale it wastes memory.

**Fix:** Delete expired entries on read, or limit the Map size.

### P2-5: `submit-feedback` Does Not Use Existing Zod Schema

**File:** `supabase/functions/submit-feedback/index.ts:23-36`

A `SubmitFeedbackBodySchema` exists in `_shared/schemas.ts` but is not imported or used. The manual validation is less robust (e.g., `tags_changed` accepts any `Record<string, unknown>` without validation, `note` has no max length check).

**Fix:** Use `SubmitFeedbackBodySchema.safeParse(rawBody)` for validation.

### P2-6: `accept-invite` Picks Arbitrary Invite for Multi-Org Users

**File:** `supabase/functions/accept-invite/index.ts:61`

`lookup.maybeSingle()` returns one row when a user has pending invites in multiple orgs. Without `org_id` in `user_metadata`, the function silently picks whichever invite `maybeSingle()` returns, which may not be the org the user intended.

**Fix:** If `invitedOrgId` is null, return a 300-style response listing pending invites for the user to choose from.

### P2-7: `create-org` — Auth Metadata Update Failure Leaves User Stateless

**File:** `supabase/functions/create-org/index.ts:136-142`

```ts
if (updateError) {
  console.error('auth metadata update error:', updateError?.message?.substring(0, 80))
}
```

If `admin.updateUserById` fails, the org and `org_members` rows are created but the user's `app_metadata.org_id` is stale. The user cannot access org-scoped features until their JWT is refreshed and the metadata is fixed.

**Fix:** Retry the metadata update, or return an error instructing the user to refresh their session.

### P2-8: `dodopayments-webhook` — Fire-and-Forget Processing

**File:** `supabase/functions/dodopayments-webhook/index.ts:157`

```ts
processEvent().catch(err => console.error(...))
```

The webhook returns 200 immediately and processes the event asynchronously. If processing fails (DB error, etc.), the payment event is silently lost with no retry mechanism.

**Fix:** Process synchronously before returning 200, or at minimum log to a dead-letter table for later retry.

### P2-9: Hardcoded `max_tokens: 12000` in Web Search

**File:** `supabase/functions/_shared/enrichment/web_search.ts:243`

Always sends `max_tokens: 12000` regardless of `maxResults`. For small requests (5 results), this allocates more output tokens than needed, increasing cost per call.

**Fix:** Scale based on `maxResults`: `Math.min(Math.max(maxResults * 300 + 1000, 2000), 12000)`.

### P2-10: Prompt Injection Surface in `personalise`

**File:** `supabase/functions/personalise/index.ts:123-144`

Prospect name, company name, and brand voice are inserted directly into the LLM prompt without sanitization. If a malicious user creates a prospect named "Ignore previous instructions...", the LLM output could be affected.

**Impact:** Low in practice — the user is only poisoning their own personalisation output.

**Fix:** Add system instructions to treat all injected data as untrusted content, or truncate/sanitize user-provided strings.

### P2-11: `llm_usage_events` Allows User UPDATE and DELETE via RLS

Migration `0012` created `org_isolation_update` and `org_isolation_delete` policies on `llm_usage_events`. This allows any authenticated member of the org to modify or delete usage records — a financial/audit table.

**Fix:** Remove UPDATE and DELETE policies. Service-role writes bypass RLS, so no user-facing mutation path is needed.

### P2-12: `increment_org_quota()` Skips NULL `asset_type`

**File:** `0006_usage_and_storage.sql`

The trigger function checks `IF NEW.asset_type IN ('image', 'carousel')` and `ELSIF NEW.asset_type = 'video'`. If `asset_type` is NULL, neither branch fires and usage is silently not tracked.

**Fix:** Add `ELSE` clause that tracks NULL asset types or add a NOT NULL constraint to the column.

### P2-13: Missing `updated_at` Triggers on Newer Tables

The `update_updated_at_column()` trigger is set up for initial tables (migration 0006) but these tables are missing it:

| Table | Has `updated_at` Column | Has Trigger |
|---|---|---|
| `org_members` | Yes | No |
| `org_linkedin_connections` | Yes | No |
| `org_chat_usage` | `last_message_at` only | N/A |
| `subscription_plans` | Yes | No |

**Fix:** Add triggers:

```sql
CREATE TRIGGER set_updated_at BEFORE UPDATE ON public.org_members
  FOR EACH ROW EXECUTE FUNCTION public.update_updated_at_column();
-- Repeat for org_linkedin_connections, subscription_plans
```

### P2-14: `feed_configs` Lacks Unique Constraint on `(org_id, source_type)`

No unique constraint prevents duplicate feed configs for the same org and source type. A bug or race condition could insert duplicates, causing duplicate signal ingestion.

**Fix:**

```sql
CREATE UNIQUE INDEX IF NOT EXISTS feed_configs_org_source_uniq
  ON public.feed_configs (org_id, source_type);
```

### P2-15: Duplicate `auth_org_id()` / `current_org_id()` Helper Functions

Two helper functions exist:

- `auth_org_id()` (migration `0002`): Simple `(auth.jwt() -> 'app_metadata' ->> 'org_id')::uuid`. No NULL handling.
- `current_org_id()` (migration `0012`): `nullif(coalesce(auth.jwt() ->> 'org_id', auth.jwt() -> 'app_metadata' ->> 'org_id'), '')::uuid`. Handles empty strings and top-level claims.

`auth_org_id()` was replaced by `current_org_id()` in migration `0012` but was never dropped. Any future migration re-creating policies might accidentally reference the old, less-safe helper.

**Fix:** Drop `auth_org_id()` after ensuring no policies reference it.

### P2-16: `orgs.plan_tier` Not Linked to `subscription_plans.tier_key`

`orgs.plan_tier` (default `'starter'`) has no foreign key or CHECK constraint referencing `subscription_plans.tier_key`. This means `orgs.plan_tier` can hold any arbitrary string, not just valid plan keys.

**Fix:** Add a foreign key or CHECK constraint.

### P2-17: `outreach_copies` NULL `platform` Breaks Unique Index

Migration `0009` creates:

```sql
CREATE UNIQUE INDEX outreach_copies_upsert_idx
  ON outreach_copies(org_id, campaign_id, prospect_id, platform)
  WHERE campaign_id IS NOT NULL;
```

Since `platform` is nullable (`text` without `NOT NULL`), two rows with the same `(org_id, campaign_id, prospect_id, NULL::text)` are allowed because NULLs are distinct in UNIQUE indexes. This undermines the intended upsert behavior.

**Fix:** Add `NOT NULL` to `platform` column or change the partial index to `WHERE campaign_id IS NOT NULL AND platform IS NOT NULL`.

### P2-18: Redundant Indexes from Migration `0011`

`0011_production_hardening.sql` creates indexes that duplicate those from `0001_initial_schema.sql`:

- `idx_campaign_briefs_org_id` duplicates `campaign_briefs_org_id_idx`
- `idx_generation_jobs_org_id` duplicates `generation_jobs_org_id_idx`
- `idx_llm_usage_events_org_id` duplicates `llm_usage_events_org_id_idx`

These are harmless (`IF NOT EXISTS` prevents errors) but waste disk and add confusion.

**Fix:** Remove the duplicate index entries from migration `0011` (or leave them — they are idempotent).

### P2-19: Overlapping Index on `signals`

- `signals_status_idx ON signals(org_id, status)` (migration `0001`)
- `signals_status_created_idx ON signals(org_id, status, created_at DESC)` (migration `0007`)

The compound index makes the simpler one redundant for typical query patterns.

**Fix:** Drop `signals_status_idx` as it is a prefix of `signals_status_created_idx`.

### P2-20: No Audit Logging for API Key Tables

No triggers or mechanisms exist to audit changes on `org_api_keys`, `org_provider_api_keys`, or `org_linkedin_connections` (which stores encrypted access tokens). A malicious actor with write access could silently swap API keys.

**Fix:** Add audit triggers that log INSERT/UPDATE/DELETE operations to a secure audit table.

### P2-21: `generation_feedback` DELETE Allows Any Org Member to Delete Others' Feedback

The `org_isolation_delete` policy on `generation_feedback` checks `org_id = auth_org_id()` but not `user_id = auth.uid()`. Any member of the org can delete any feedback row, not just their own.

**Fix:** Add `AND user_id = auth.uid()` to the DELETE policy, or remove the DELETE policy entirely if feedback should be immutable.

## 8. Post-Audit Fixes (2026-05-25/26)

### P0-0f: LinkedIn chat "not connected" despite valid connection
- **Root cause:** Two bugs:
  1. `fetchLinkedInSnapshot()` used `.single()` which throws when no row exists, and caught token errors silently returning `''`. The chat LLM then assumed LinkedIn was disconnected.
  2. RLS policy on `org_linkedin_connections` used `auth.jwt() -> 'app_metadata' ->> 'org_id'` which is stale on first JWT load, causing the frontend query to return null on first page load ("LinkedIn not connected" banner flickered, then showed "LinkedIn live" after first chat message).
- **Fix:**
  - `campaign-chat/index.ts`: Changed `fetchLinkedInSnapshot` return type to `{ snapshot: string; connected: boolean }`. Even if the LinkedIn API call fails (401, 403, no ad data, network error), `connected: true` is returned when a row exists in `org_linkedin_connections`. The system prompt now has three states: full analytics snapshot, "connected but no analytics", or "not connected".
  - `campaign-chat.tsx`: Changed `linkedinConnected` state from `boolean` (default `false`) to `boolean | null` (default `null`). Shows "Checking LinkedIn..." loading state instead of "LinkedIn not connected" while the initial query resolves.
  - `org_linkedin_connections` RLS policies: Changed from `auth.jwt() -> 'app_metadata' ->> 'org_id'` to `current_org_id()` (same as other org-scoped tables). Also fixed `campaign_chat_messages` and `org_chat_usage` RLS policies.
  - `org_linkedin_connections` table: Made `ad_account_urn` nullable (was `NOT NULL`) since LinkedIn connections can be valid without an ad account. Updated CHECK constraint accordingly.
  - `save-linkedin-connection/index.ts`: Changed `ad_account_urn: adAccountUrn` to `ad_account_urn: adAccountUrn || null` to store NULL instead of empty string.
- **Migration:** `20260526000000_0031_linkedin_connections_rls_fix.sql` (needs manual application via Supabase Dashboard SQL Editor)
- **Status:** Code deployed (edge functions + frontend). **Migration pending manual application.**
